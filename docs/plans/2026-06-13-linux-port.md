# DeepSteps → Linux VST3/CLAP Plugin

## Context

DeepSteps is a standalone macOS openFrameworks app: a MIDI step sequencer whose
patterns are generated by a user-trainable autoencoder. Goal: make it run as a
Linux x86_64 VST3/CLAP plugin.

Key architectural facts found during exploration (cite for the port):
- **Sequencer brain lives in the Pd patch** (`bin/data/pd/main.pd`), not C++.
  Pd does clock counting (192 PPQN, `mod 192`), step advance, step→note mapping,
  key/scale quantization, and substep timing offsets. C++ (`ofApp.cpp`) is ~31%
  GUI drawing + MIDI plumbing + Python embedding.
- **Clock** = external MIDI clock in (`ofxMidiClock`, `ofApp.cpp:1199-1287`). No
  internal clock.
- **Runtime ML flow**: sliders → C++ builds Python string → `PyRun_SimpleString`
  (`ofApp.cpp:292-307`) → `ae1.decoder.predict` (`AE_init.py:383-402`) → OSC to
  `127.0.0.1:9000` → Pd unpacks `/sequence` + `/steps` → drives steps →
  `noteout` → `receiveNoteOn` (`ofApp.cpp:1170`) → `midiOut.sendNoteOn`.
- **aubio** used only offline (corpus/dataset build, `ofApp.cpp:778-874`).
- Hardcoded macOS-isms that break on Linux/plugin: `system("open ...")`
  (`:772`), `$HOME/data` + working-dir assumptions (`:42-49`), absolute font/
  script paths, OSC port 9000.

### Decisions (locked with user)
- **Plugin role:** MIDI generator only — host makes sound. Drop Pd *audio synth*.
- **ML:** training stays offline in Python; plugin only loads a trained model and
  generates.
- **Path:** staged. **Stage 1 = Linux standalone first**, **Stage 2 = plugin**.
- **Stage 1 scope:** runtime only — skip aubio + in-app dataset build/training UI.
- **Stage 2 framework:** **nih-plug (Rust)** — full rewrite, CLAP + VST3.

### Important consequence
Stage 2 (nih-plug) reuses **none** of the C++/OF/Pd code. So Stage 1's job is to
be a **working behavioral reference** on Linux: prove the pipeline, then extract
exact algorithms (Pd sequencer math, decoder forward pass, scale tables) and A/B
the Rust plugin's MIDI output against it. If standing up OF on Linux proves too
costly (GCC 16 vs old OF), we can pivot to extracting the logic by reading the Pd
patch + Python directly and skip the standalone build.

## Stage 1 — Linux x86_64 standalone (runtime-only)

Deliverable: the existing app builds with `make` and runs on this machine,
loading a pre-trained model and emitting MIDI notes when clocked.

Toolchain check already done: OF **not installed**; ALSA dev 1.2.16 ✓; g++ 16.1
+ cmake 4.3 ✓; aubio/pd not installed (ofxPd bundles libpd — system pd not
needed); Python3 present behind `uv`; git remote is upstream `ajwast/DeepSteps`.

### Steps
1. **Install openFrameworks (Linux x86_64 release).** Match the version the macOS
   project expects (inspect `Deep Steps.xcodeproj/project.pbxproj` /
   `Project.xcconfig` for hints; likely 0.11.x/0.12.x). Run OF's
   `scripts/linux/download_libs.sh` + `install_dependencies.sh`. **Risk:** GCC 16
   is far newer than these OF releases — expect compile patches; pin to OF 0.12+
   if possible.
2. **Place addons** in `OF_ROOT/addons/`: `ofxGui` (core), `ofxMidi`, `ofxPd`,
   `ofxSimpleSlider`. ofxMidi → ALSA/RtMidi backend; ofxPd → builds libpd.
3. **Point the project at Linux OF:** rewrite `Deep_Steps_project/config.make`
   (currently hardcoded `/Users/alexwastnidge/Desktop/openFrameworks`) to the
   Linux `OF_ROOT`. Generate/confirm Linux makefiles via OF projectGenerator or
   the standard `Makefile` + `config.make` pair.
4. **De-macOS the source** (`src/ofApp.cpp`):
   - Replace `system("open ...")` (`:772`) — guard out (it's in the dataset path
     we're skipping anyway) or use `xdg-open`.
   - Fix data-path resolution (`:42-49`, font `:56`, script loads `:64/177/845/884`)
     to use `ofToDataPath()` / relative `bin/data`, no `$HOME/data`.
5. **Disable the skipped features:** stub the "Make Dataset" / aubio path
   (`:778-874`) and, if it pulls aubio, the corpus button, so the app links
   without aubio. Keep `trainButton` only if trivially Python-only (AE_train.py is
   4 lines, no aubio) — otherwise stub it too. Ship a **pre-trained model +
   `dataset.csv` + `midiCCmap.txt`** in `bin/data` so generation works.
6. **Embedded Python on Linux:** link `libpython3.x`; ensure the embedded
   interpreter finds `python-osc`, `numpy`, and the ML-From-Scratch autoencoder
   modules (provide a venv on `PYTHONPATH`, or system site-packages). The runtime
   path needs `receive_latent` → `decoder.predict` → OSC 9000 → Pd to work.
7. **Build:** `make -C Deep_Steps_project` → `bin/Deep Steps`.

### Stage 1 verification
- Launch the binary; it should open the GL window without crashing.
- Create ALSA MIDI plumbing: a virtual clock/transport source + a monitor
  (`aconnect -l`, `aseqdump -p <port>`), connect app's MIDI in/out.
- Send MIDI clock + start; confirm steps advance and **note-on/off appear in
  `aseqdump`**.
- Click **Generate**; confirm the pattern changes (decoder → OSC → Pd → new
  notes). This proves the full runtime pipeline on Linux.
- Record a reference MIDI capture for Stage 2 A/B comparison.

## Stage 2 — nih-plug (Rust) VST3 + CLAP plugin (outline; detail-plan after Stage 1)

Full rewrite. Reuses no C++. Plan in detail once Stage 1 gives a reference.

### Shape
- New Rust crate using **nih-plug**, exporting **CLAP + VST3** via
  `nih_export_clap!` / `nih_export_vst3!`, built with the nih-plug xtask bundler.
- **MIDI generator plugin:** consumes host transport (tempo + PPQ position) for
  clock instead of MIDI-clock-in; emits `NoteEvent::NoteOn/NoteOff` per step.
- **Re-derive the sequencer from the Pd patch** into Rust: 192-PPQN step counter,
  16-step loop, step→pitch, key/scale quantization tables, substep timing offset,
  `seqLength`, `gateLength`.
- **Decoder inference in Rust:** offline Python export of trained autoencoder
  **decoder weights** to a file; implement the forward pass (dense layers +
  activation) and the 0.5 step threshold + substep split (mirror
  `AE_init.py:383-402`). No embedded Python, no OSC, no Pd at runtime.
- **Parameters** (from the ofxGui widgets): tempo (or host-synced), 16 note
  pitches, latent A–D, gate length, substeps scale, seq length, key, scale, and
  the CC→latent map. Expose as nih-plug `Param`s; wire host MIDI CC to latent.
- **UI:** nih-plug `egui` (or `vizia`) editor; port the 16-step visualization.
- **Drop:** Pd, embedded Python runtime, OSC port 9000, aubio.

### Stage 2 verification (later)
- `cargo xtask bundle` produces `.clap` + `.vst3`; validate with
  `clap-validator` and a host (Bitwig/Carla/Reaper on Linux).
- Load in Carla, drive from host transport, confirm note output in `aseqdump`.
- **A/B against the Stage 1 reference capture** for identical params/seed.

## Risks
- **OF on GCC 16:** old OF releases may not compile cleanly; budget patch time or
  pin OF version. This is the biggest Stage-1 unknown.
- **Embedded Python deps** (numpy / ML-From-Scratch / python-osc) discoverable by
  the in-process interpreter on Linux.
- **Pd → Rust fidelity:** the patch encodes the real timing/quantization; porting
  must be exact — hence the Stage 1 reference capture.
- **Decoder weight export:** need a stable serialization the Rust forward pass
  reads; verify numerically against Python.
- Pushing later needs your own fork (remote is upstream `ajwast/DeepSteps`).
